What is FinOps Agent?

An AI-powered AWS cost optimization assistant built using Amazon Bedrock AgentCore. The agent provides intelligent cost analysis, forecasting, and optimization recommendations through natural language conversations.

Key Features

  • 20 Cost Optimization Tools - Comprehensive billing and pricing analysis
  • Gateway Architecture - Scalable MCP Lambda server integration
  • Memory Integration - 30-day conversation context retention
  • Cognito Authentication - Secure user access control
  • CDK-Nag Compliant - Security best practices enforced
  • React Frontend - Modern chat interface with Amplify v6

2. Architecture

Architecture Diagram

Complete architecture showing all AWS services and data flow

High-Level Architecture

User → AWS Amplify → Cognito → AgentCore Runtime (ECS) → Gateway → MCP Lambda Servers ├─ Billing Lambda (11 tools) └─ Pricing Lambda (9 tools)

Architecture Layers

Layer 1: User Interface

  • AWS Amplify - Hosts React frontend application
  • React Chat Interface - Markdown support, real-time responses

Layer 2: Authentication

  • Amazon Cognito User Pool - User authentication
  • Amazon Cognito Identity Pool - AWS credentials for authenticated users

Layer 3: Agent Core Runtime

  • Amazon ECS Fargate - Runs containerized Python agent
  • Bedrock AgentCore Runtime - Agent execution environment
  • Bedrock AgentCore Memory - Conversation history (30-day retention)
  • Bedrock AgentCore Gateway - Routes tool requests to Lambda

Layer 4: Tool Execution

  • Billing MCP Lambda - 11 tools for cost analysis
  • Pricing MCP Lambda - 9 tools for pricing information

Layer 5: Data Sources

  • AWS Cost Explorer - Historical cost and usage data
  • AWS Budgets - Budget management
  • AWS Compute Optimizer - Resource optimization recommendations
  • AWS Free Tier API - Free tier usage tracking
  • AWS Pricing API - Service pricing information

Layer 6: Supporting Services

  • Amazon ECR - Container image storage
  • AWS CodeBuild - ARM64 Docker image build
  • AWS IAM - Permissions and roles
  • Amazon CloudWatch Logs - Monitoring and logging

6. Deployment Process

Prerequisites

  • AWS CLI configured with credentials
  • Node.js 18+ and npm
  • Python 3.13+
  • Admin email address for Cognito user

Deployment Steps

# 1. Clone repository git clone https://github.com/aws-samples/sample-finops-agent-amazon-agentcore.git cd sample-finops-agent-amazon-agentcore # 2. Run interactive deployment ./scripts/deploy-interactive.sh # The script will: # - Prompt for admin email # - Show deployment summary # - Install CDK dependencies (npm install) # - Build TypeScript (npm run build) # - Bootstrap CDK if needed # - Deploy all 3 stacks in order # - Display outputs (Runtime ARN, etc.)

Deployment Sequence

  1. FinOpsImageStack (5-10 minutes)
    • Creates ECR repository
    • Creates CodeBuild project
    • Builds ARM64 Docker image
    • Pushes image to ECR
  2. FinOpsAgentStack (10-15 minutes)
    • Creates IAM roles
    • Creates Lambda functions
    • Creates Gateway with targets
    • Creates Memory
    • Creates Runtime (ECS Fargate)
  3. FinOpsAuthStack (2-3 minutes)
    • Creates Cognito User Pool
    • Creates admin user
    • Sends temporary password email
    • Creates Identity Pool
Total Deployment Time: ~20-30 minutes

7. Challenges & Solutions

Challenge 1: Gateway Permission Issue

Problem: Gateway targets failing with "Gateway execution role lacks permission to invoke Lambda function"
Root Cause: Gateway targets were being created before the Gateway's service role IAM policy was fully created, causing a CloudFormation dependency/timing issue.
Solution: Added explicit CloudFormation dependencies to ensure Gateway service role policy is created before targets:
// Capture target return values const billingTarget = gateway.addLambdaTarget('BillingTarget', {...}); const pricingTarget = gateway.addLambdaTarget('PricingTarget', {...}); // Add explicit dependencies const gatewayServiceRole = gateway.node.findChild('ServiceRole') as iam.Role; const gatewayServiceRolePolicy = gatewayServiceRole.node.findChild('DefaultPolicy') as iam.Policy; billingTarget.node.addDependency(gatewayServiceRolePolicy); pricingTarget.node.addDependency(gatewayServiceRolePolicy);

Challenge 2: Memory Integration Bug

Problem: Agent runtime crashing with AttributeError: 'Agent' object has no attribute 'tools'
Root Cause: Agent runtime was trying to access agent.tools and agent.system_prompt which don't exist in the Strands Agent object.
Solution: Store mcp_tools and system_prompt_template as global variables instead of trying to access them from the agent object:
# Store as globals mcp_tools = [] system_prompt_template = "" # Use globals in memory retrieval memory_context = f"Previous conversation:\n{memory_data}\n\nAvailable tools: {mcp_tools}\n\nSystem: {system_prompt_template}"

Challenge 3: Frontend Response Parsing

Problem: Frontend showing "Sorry, I couldn't process your request" despite receiving valid response
Root Cause: Backend was returning {message: {...}} but frontend expected {result: "..."}
Solution: Changed backend response format to match frontend expectations (simpler than modifying frontend for 3 agent types)

Challenge 4: Amplify v6 Migration

Problem: Frontend using old Amplify v5 syntax with Amplify v6 package installed
Root Cause: Amplify v6 has breaking changes in import syntax
Solution: Updated imports and function calls to Amplify v6 syntax:
// Old (v5) import * as AWSAuth from '@aws-amplify/auth'; await AWSAuth.fetchAuthSession(); // New (v6) import { fetchAuthSession, signOut } from 'aws-amplify/auth'; await fetchAuthSession();

8. CDK-Nag Compliance

What is CDK-Nag?

CDK-Nag is a tool that checks CDK applications for security and compliance best practices. It identifies potential security issues and provides recommendations.

Implementation

// Added to cdk/bin/app.ts import { AwsSolutionsChecks } from 'cdk-nag'; import { Aspects } from 'aws-cdk-lib'; const app = new cdk.App(); Aspects.of(app).add(new AwsSolutionsChecks({ verbose: true }));

Security Fixes Applied

Rule ID Issue Action
AwsSolutions-S10 S3 bucket without SSL enforcement FIXED - Added enforceSSL: true
AwsSolutions-COG1 Cognito password policy missing symbols FIXED - Changed requireSymbols: true

Suppressions Applied

Rule ID Reason for Suppression

AwsSolutions-IAM4 AWS managed policies required for Lambda and AgentCore service roles

AwsSolutions-L1 Python 3.13 is the latest Lambda runtime version
AwsSolutions-CB4 CodeBuild KMS encryption not required for public Docker image build
AwsSolutions-COG2 Cognito MFA not required for demo/development
AwsSolutions-COG3 Cognito advanced security not required for demo/development

Lessons Learned

  1. CloudFormation Dependencies Matter - Explicit dependencies prevent timing issues
  2. Gateway Uses Its Own Role - Don't create custom roles, use gateway.grantPrincipal
  3. Memory Requires Careful Handling - Store context as globals, not in agent object
  4. CDK-Nag Suppressions Need Justification - Document why wildcards are necessary
  5. Amplify v6 Has Breaking Changes - Update imports when migrating
  6. Test Early and Often - Catch issues before they compound
